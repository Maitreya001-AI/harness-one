01一句话定义

Harness Engineering 是围绕 AI Agent 设计环境、约束和反馈回路的工程学科，使 Agent 在生产环境中可靠运行。公式：Agent = Model + Harness。Model 提供智能，Harness 提供一切其他——工具、记忆、状态、权限、验证、生命周期管理、编排、熵回收。

⚙
模型 = CPU
原始算力。你不写汇编操控寄存器，同理你不直接 prompt 裸模型做复杂生产任务。
⟦⟧
Context = RAM
有限、易失的工作记忆。Token 窗口就是内存——有注意力预算、会 context rot、满了得 swap。
◈
Harness = OS
操作系统。管理上下文、调度工具、权限控制、启动引导、标准驱动、可观测性、熵回收。
▲
Agent = 应用
跑在 OS 上的应用程序。你的业务技能、对话交互、任务编排就是应用层。
核心洞察：Harness 比 Model 更重要。LangChain DeepAgents 仅换 Harness（模型不变），TerminalBench 分数 52.8% → 66.5%。Anthropic 在 SWE-bench 上定制 Harness 多带来 10 个百分点。OpenAI 用 Codex + Harness，零手写代码，5 个月 100 万行生产代码。
02学科层级

Harness Engineering 在新兴学科体系中的位置——每一层包含上面所有层，但增加新的关注维度：

1
Prompt Eng.
优化单次模型交互——写好一个 prompt。
2
Context Eng.
管理工作记忆预算——单次调用内模型看到什么。包含 Prompt Eng.。
3
Agent Eng.
优化 Agent 决策逻辑——循环、工具选择、目标追踪。包含 Context Eng.。
4
Harness Eng.
设计整个系统——工具、约束、反馈回路、状态、治理、生命周期、可观测、熵回收。包含 Agent Eng.。
Martin Fowler 的另一个分析维度：Harness 的控制手段可以按两个正交轴分类——方向（Feedforward 前馈 vs Feedback 反馈）和 执行（Computational 确定性 vs Inferential 推理性）。前馈引导 Agent 做对，反馈在 Agent 做错后纠正。确定性手段（Linter、测试）便宜可靠，推理性手段（LLM-as-judge）昂贵但能处理语义问题。
039 层参考架构

这不是行业标准（目前没有统一标准），而是综合所有主要来源后的一个参考模型。行业里已有的分类法包括 harness-engineering.ai 的 6 组件模型、Martin Fowler 的 Feedforward/Feedback × Computational/Inferential 矩阵、以及学术界的 CAR（Control-Agency-Runtime）分解。本模型试图以「实现层」的视角将它们统一。

不是说必须全部实现——Phil Schmid 的第一原则就是「Start Simple」。但你需要知道完整蓝图，才能有意识地选择做什么、不做什么。

075 级成熟度模型

用来评估你的 Agent 产品当前处于 Harness Engineering 的哪个阶段，以及下一步该做什么：

L0
裸奔
直接调 LLM API，无护栏、无可观测性、无工具限制。Demo 能跑，生产必崩。大多数 hackathon 项目在这里。
L1
基础 Harness
有 Agent Loop（ReAct/while-true）、基础工具系统（声明式定义 + 参数校验）、简单的错误处理。能处理 happy path，但长对话、边界情况和安全性未覆盖。
L2
结构化 Harness
加入上下文工程（预算分配 + Lost-in-the-Middle 布局）、输入/输出护栏、结构化记忆（跨会话持久化）、基础可观测性（trace + cost）。可以上生产，但质量波动大、缺少闭环改进机制。
L3
可度量 Harness
加入 Eval 评分器 + CI 门禁（质量可量化）、Generator-Evaluator 分离（质量有兜底）、KV-cache 命中率监控（成本可优化）、渐进式技能披露（上下文可控）、跨 session 进度接力。质量稳定，能做 A/B 测试和持续改进。
L4
自演进 Harness
品味编码飞轮制度化（失败 → 规则 → 自动化 → 度量）、Build to Delete 自动化（组件拦截率 → 退役 Issue）、自动化熵回收（后台偏差扫描 + 修复 PR）、Harness-as-Dataset（轨迹数据反哺模型训练）。Harness 自己能变好，不需要人全程盯着。

09三个反直觉发现
① Harness 复杂度应与模型能力反向变化。模型越强，最优 Harness 越简单。Anthropic 在 Opus 4.6 发布后移除了三 Agent 架构的 sprint 机制。每个 Harness 组件都编码了一个「模型做不到 X」的假设——这些假设要持续压力测试。
② 约束增加可靠性。不是给 Agent 更多自由让它变好——而是设计正确的边界。Vercel 移除 80% 的工具反而提升了表现。Manus 用 Tool Masking 而不是移除工具。OpenAI 用 Linter 强制分层。
③ 协同进化反馈循环。前沿模型正在针对其原生 Harness 做后训练（Claude ↔ Claude Code，GPT ↔ Codex）。模型在「自家」Harness 中的表现可能显著优于第三方。第三方 Harness 需要在工程层面弥补这一差距。
