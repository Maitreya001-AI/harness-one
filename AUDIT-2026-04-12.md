## Category 1: Logic Errors / Bugs

### [BUG-01] CRITICAL — content-filter: Custom patterns with `g` flag cause intermittent detection failures

**File:** `packages/core/src/guardrails/content-filter.ts:86-88`

The `blockedKeywords` loop correctly resets `lastIndex` (line 80), but custom `blockedPatterns` do NOT:

```typescript
// Line 78-83: blockedKeywords CORRECTLY reset lastIndex
for (const keyword of blockedKeywords) {
  keyword.regex.lastIndex = 0;  // ← correct
  if (keyword.regex.test(contentNormalized)) { ... }
}

// Line 86-88: custom patterns DO NOT reset lastIndex
for (const pattern of patterns) {
  if (pattern.test(ctx.content)) { ... }  // ← BUG: no lastIndex reset
}
```

**Impact:** If a user provides a pattern with the `g` flag, `pattern.test()` advances `lastIndex` on each invocation. On the second call, matching starts mid-string, causing intermittent false negatives. This is a security-critical guardrail bypass.

**Fix:** Add `pattern.lastIndex = 0;` before `pattern.test()`.

---

### [BUG-02] HIGH — OpenAI adapter: Zero-token fallback corrupts cost tracking

**File:** `packages/openai/src/index.ts:307-311`

```typescript
// Only emit bare done if stream ended without a usage chunk
yield {
  type: 'done' as const,
  usage: { inputTokens: 0, outputTokens: 0 },
};
```

**Impact:** If the OpenAI API changes behavior or a compatible provider (Groq, DeepSeek, etc.) doesn't include `stream_options: { include_usage: true }`, ALL cost tracking reports $0. This silently corrupts billing data with no warning.

**Fix:** Log a warning when falling back to zero tokens. Consider accumulating tokens from stream deltas as a fallback estimate.

---

### [BUG-03] HIGH — cache-stability: `messageKey()` produces `[object Object]` for array content

**File:** `packages/core/src/context/cache-stability.ts:110-111`

```typescript
function messageKey(m: Message): string {
  return `${m.role}::${m.content}::${m.name ?? ''}`;
}
```

**Impact:** `Message.content` can be a `ContentBlock[]` (array of objects). Template literal coercion produces `"user::[object Object],[object Object]::"`—all multi-block messages hash to the same key, making overlap ratio meaningless.

**Fix:** Use `JSON.stringify(m.content)` instead of implicit coercion.

---

### [BUG-04] MEDIUM — content-filter: Pattern tested against raw content instead of normalized content

**File:** `packages/core/src/guardrails/content-filter.ts:87`

```typescript
if (pattern.test(ctx.content)) {  // raw content
```

But `blockedKeywords` are tested against `contentNormalized` (NFKC + lowercase, line 81). Custom patterns bypass normalization entirely, meaning Unicode confusables and case variations evade custom pattern matching.

**Fix:** Apply `pattern.test(contentNormalized)` or document that patterns receive raw input.

### [BUG-05] MEDIUM — checkpoint: No bounds validation on `maxCheckpoints`

**File:** `packages/core/src/context/checkpoint.ts:65,69-73`

```typescript
const maxCheckpoints = config?.maxCheckpoints ?? 5;
// ...
function autoPrune(): void {
  const list = storage.list();
  if (list.length >= maxCheckpoints) {  // always true if maxCheckpoints <= 0
    storage.delete(list[0].id);
  }
}
```

**Impact:** `maxCheckpoints = 0` causes `autoPrune()` to delete a checkpoint on every `save()`, effectively disabling checkpointing silently. Negative values also accepted.

**Fix:** Validate `maxCheckpoints >= 1` at construction.

## Category 2: Performance Bottlenecks

### [PERF-01] CRITICAL — OpenTelemetry: O(n log n) sort on every span eviction

**File:** `packages/opentelemetry/src/index.ts:89-121`

```typescript
function evictSpans(count: number): void {
  // Sort ALL spans by lastAccessTime ascending
  const sortedEntries = [...spanAccessTime.entries()].sort((a, b) => a[1] - b[1]);
  // ...
  if (evictedParents.size > maxEvictedParents) {
    const evictedSorted = [...evictedParentsAccessTime.entries()].sort(...);  // ANOTHER sort
  }
}
```

**Impact:** With `maxSpans = 10,000`, every span insertion that triggers eviction copies and sorts 10,000 entries (O(n log n)), then potentially sorts the evicted parents map too. Under high-throughput tracing (100+ spans/sec), this causes event-loop pauses of 10-50ms.

**Fix:** Replace with a proper LRU using Map insertion-order (delete-then-reinsert pattern, like Langfuse already does). Alternatively, use a min-heap for O(log n) eviction.

---

### [PERF-02] HIGH — OpenTelemetry: O(n) stale parent purge on every export

**File:** `packages/opentelemetry/src/index.ts:66-73`

`purgeStaleEvictedParents()` iterates the entire `evictedParentsAccessTime` map on every `getEvictedParentSpan()` call, which is called during every span export.

**Impact:** With many evicted parents, this adds O(n) overhead to every single span export operation.

**Fix:** Use lazy TTL expiry (check on access only, not scan-all) — the same pattern used in session manager.

---

### [PERF-03] HIGH — cost-tracker: O(n) `getCostByTrace()` with no index

**File:** `packages/core/src/observe/cost-tracker.ts:341-345`

```typescript
getCostByTrace(traceId: string): number {
  return records.filter(r => r.traceId === traceId)
    .reduce((sum, r) => sum + r.estimatedCost, 0);
}
```

**Impact:** Scans all 10,000 records (maxRecords) per call. In dashboards or monitoring that poll trace costs, this causes latency spikes.

**Fix:** Maintain a `Map<traceId, KahanSum>` secondary index (same pattern as `modelTotals`).

---

### [PERF-04] MEDIUM — orchestrator: O(n²) cycle detection via `queue.shift()`

**File:** `packages/core/src/orchestration/orchestrator.ts:294`

BFS cycle detection uses `queue.shift()` in a loop, which is O(n) per shift on arrays. Total complexity is O(n²) for the queue operations alone.

**Fix:** Use an index pointer instead of shift, or use a proper deque.

---

### [PERF-05] MEDIUM — memory store: Unconditional sort on every query

**File:** `packages/core/src/memory/store.ts:256`

```typescript
results.sort((a, b) => b.updatedAt - a.updatedAt);
```

Every `query()` call sorts results even when the caller doesn't need ordered results. With many entries, this adds O(n log n) overhead per query.

**Fix:** Make sorting opt-in via a `sortBy` filter option.

---

### [PERF-06] MEDIUM — fs-store: Loads all entries before filtering

**File:** `packages/core/src/memory/fs-store.ts:98-116`

Batch reads all entry files, deserializes them all to JSON, THEN applies filters. For large stores, this loads the entire dataset into memory.

**Fix:** Store a lightweight index file with grade/tags/timestamps so filtering can happen before deserialization.

## Category 3: Resource Management

### [RES-01] HIGH — cost-tracker: `modelTotals` grows unboundedly

**File:** `packages/core/src/observe/cost-tracker.ts:117-123`

```typescript
const modelTotals = new Map<string, KahanSum>();
```

New model entries are added permanently, never evicted. In systems that dynamically generate model identifiers (e.g., fine-tuned model IDs), this map grows without bound.

**Fix:** Add a `maxModels` limit with LRU eviction, or document that model identifiers must be a finite set.

---

### [RES-02] HIGH — full package: `drain()` doesn't clean up memory/session resources

**File:** `packages/full/src/index.ts:420-436`

The `drain()` method aborts the loop, flushes traces, and calls `shutdown()`. But it never:
- Calls `sessions.dispose()` (which would clear the GC interval timer)
- Closes Redis connections if using `@harness-one/redis`
- Disposes the memory store

**Impact:** In serverless or test environments, leaked timers and connections prevent clean process exit.

**Fix:** Add `sessions.dispose()` and optional `memory.dispose?.()` calls to shutdown/drain.

---

### [RES-03] HIGH — context-boundary: View cache leaks on agent removal

**File:** `packages/core/src/orchestration/context-boundary.ts:54`

The `viewCache` Map stores views per agent. When agents are unregistered from the orchestrator, their cached views are never cleaned up.

**Impact:** In long-running orchestration with many ephemeral agents, memory grows linearly with total agents ever created.

**Fix:** Add a `clearAgent(agentId)` method or hook into orchestrator's agent removal lifecycle.

---

### [RES-04] HIGH — OpenTelemetry: `flush()` clears state during concurrent exports

**File:** `packages/opentelemetry/src/index.ts:243-265`

```typescript
async flush(): Promise<void> {
  // Migrates all active spans to evictedParents, then clears spanMap
  spanMap.clear();  // line 262
}
```

**Impact:** If `exportSpan()` is called concurrently with `flush()`, the span being exported may be deleted from spanMap mid-operation. New spans added during flush are also lost.

**Fix:** Use a generation counter or snapshot-then-clear pattern to isolate the flush operation.

---

### [RES-05] MEDIUM — Langfuse: `maxRecords` not validated

**File:** `packages/langfuse/src/index.ts:321-323`

If `maxRecords` is set to 0 or negative, the records array grows without bound (the eviction check `records.length > maxRecords` is always true but `records.shift()` creates an infinite-shift loop, or never true for negative).

**Fix:** Validate `maxRecords >= 1` at construction.

---

### [RES-06] MEDIUM — agent-pool: `monotonicNow()` uses `Date.now()` despite claiming monotonic

**File:** `packages/core/src/orchestration/agent-pool.ts:58-64`

The comment says `performance.now()` is preferred but `Date.now()` is used for test compatibility. This is documented but the function name `monotonicNow` is misleading—`Date.now()` is NOT monotonic and is affected by NTP corrections and clock drift.

**Fix:** Rename to `now()` to avoid implying monotonic guarantees, or use `performance.now()` with a test-injection mechanism.

## Category 4: Error Handling Gaps

### [ERR-01] HIGH — compress: Summarizer failure crashes without fallback

**File:** `packages/core/src/context/compress.ts:231`

```typescript
const summary = await summarizer(toSummarize);
```

**Impact:** If the LLM-based summarizer throws (rate limit, network error, malformed response), the error propagates unhandled. No fallback to truncation strategy. In production, this can break the entire conversation flow.

**Fix:** Wrap in try-catch with fallback to truncation:
```typescript
let summary: string;
try {
  summary = await summarizer(toSummarize);
} catch {
  // Fallback: use last N messages as context
  return messages.slice(-Math.ceil(messages.length / 2));
}
```

---

### [ERR-02] HIGH — full package: `conversations.append()` errors silently swallowed

**File:** `packages/full/src/index.ts:337,376,395,399`

```typescript
await conversations.append('default', msg);  // No try-catch
```

Four separate `conversations.append()` calls lack error handling. If the underlying store fails (e.g., Redis timeout), conversation history is silently lost.

**Fix:** Wrap in try-catch with logger.warn, or propagate as a non-fatal warning event.

### [ERR-03] HIGH — full package: Exporter shutdown errors silently swallowed

**File:** `packages/full/src/index.ts:405-418`

```typescript
await Promise.race([
  exporter.shutdown(),  // If this throws, it's silently lost
  new Promise<void>((resolve) => setTimeout(resolve, EXPORTER_TIMEOUT)),
]);
```

**Impact:** If `exporter.shutdown()` rejects, the rejection is unhandled (no `.catch()`). This can trigger Node.js `unhandledRejection` warnings or crash the process with `--unhandled-rejections=throw`.

**Fix:** Add `.catch(err => logger.warn('Exporter shutdown error', { error: err }))` to the shutdown promise.

**File:** `packages/core/src/core/output-parser.ts:98-117`

The `regenerate()` callback is awaited without timeout. If the LLM regeneration hangs, parsing hangs indefinitely per retry attempt.

**Fix:** Add an optional `regenerateTimeoutMs` parameter with a sensible default (e.g., 30s).

Also at lines 149-150, 153-154. Real network errors, authentication failures, and data corruption are all invisible.

### [ERR-09] MEDIUM — env.ts: `Infinity` and `NaN` not caught by validation

## Category 5: Robustness / Edge Cases

### [ROB-01] HIGH — Redis store: Non-atomic read-modify-write in `update()`

**File:** `packages/redis/src/index.ts:164-182`

```typescript
async update(id, changes) {
  const existing = await read(id);  // Step 1: Read
  if (!existing) return;
  const updated = { ...existing, ...changes };  // Step 2: Modify
  await write(updated);  // Step 3: Write
}
```

**Impact:** Between read and write, another process can modify or delete the entry. The update overwrites their changes (lost update problem).

**Fix:** Use Redis WATCH/MULTI/EXEC for optimistic locking, or Lua scripting for atomic updates.

---

### [ROB-02] HIGH — session manager: LRU eviction fails silently when all sessions are locked

**File:** `packages/core/src/session/manager.ts:138-148`

When `maxSessions` is reached and all existing sessions are locked, eviction silently fails and the new session is created anyway, violating the `maxSessions` contract.

**Fix:** Throw an error or return a "session limit reached" result when eviction is impossible.

### [ROB-09] LOW — checkpoint: ID generation may collide under rapid creation

## Category 6: Architecture & Missing Features

### [ARCH-01] Event Bus is deprecated but still wired into `createHarness()`

**File:** `packages/core/src/core/event-bus.ts` (deprecated), `packages/full/src/index.ts:313`

The EventBus is marked `@deprecated` three times but is still created and exposed in the full harness factory. No removal timeline is specified.

**Recommendation:** Either remove it entirely or set a deprecation timeline (e.g., "removed in v0.2.0").

---

### [ARCH-02] No streaming backpressure mechanism

The agent loop's `run()` is an `AsyncGenerator` that yields events. If the consumer is slow (e.g., writing to a database per event), the producer continues unthrottled. There's no backpressure signal.

**Recommendation:** Consider a bounded channel or async queue between producer and consumer, or document that consumers must be fast.

---

### [ARCH-03] No graceful in-flight tool call completion on drain

**File:** `packages/full/src/index.ts:420-436`

`drain()` calls `loop.abort()` immediately, which sends an AbortSignal to running tool calls. But it doesn't wait for in-flight tool calls to finish — it just sleeps 100ms and hopes.

**Recommendation:** Track in-flight tool calls and await their completion with a deadline before shutting down.

---

### [ARCH-04] No health check / readiness probe

There's no way for orchestration or deployment systems to check if a harness instance is healthy (adapter reachable, memory store connected, exporters operational).

**Recommendation:** Add a `harness.healthCheck()` method that probes all subsystems.

---

### [ARCH-05] No metrics export (Prometheus / StatsD)

Cost tracking, cache hit rates, and failure taxonomy data are tracked internally but have no export path to standard monitoring systems.

**Recommendation:** Add a `createMetricsExporter()` interface with Prometheus and StatsD implementations.

---

### [ARCH-06] fs-store only has in-process mutex — unsafe for multi-process

**File:** `packages/core/src/memory/fs-store.ts:19-22`

The file-system store uses an in-process lock (`withIndexLock`). Multiple Node.js processes accessing the same directory will corrupt the index.

**Recommendation:** Use file-based locking (`fs.flock` or `proper-lockfile`) or document this limitation prominently.

---

### [ARCH-07] No configuration hot-reload

All configuration is set at construction time. Changing guardrail rules, cost budgets, or pricing requires restarting the harness.

**Recommendation:** Add a `harness.reconfigure(partial)` method for mutable configuration sections.

---

### [ARCH-08] No plugin/extension system

All integrations are compile-time dependencies. There's no runtime plugin loading mechanism for custom guardrails, exporters, or memory backends without importing them at build time.

**Recommendation:** Define a `HarnessPlugin` interface with lifecycle hooks (init, shutdown) and a `harness.use(plugin)` registration pattern.
