### Merge: `Message` ← { `Message`, `SystemMessage`, `UserMessage`, `AssistantMessage`, `ToolMessage`, `MessageParam`, `ChatCompletionMessageParam` }

**Judgment 2 application**:
- Lifecycle test: same — all are immutable-after-creation, all may be
  appended to a conversation, all may be persisted to a Session or
  MemoryEntry, all may be downgraded on restore (system specifically).
- Permission test: same — no per-role ownership; conversations are
  application-controlled; trust is captured by the SystemMessage `_trust`
  brand, which is a *field* not a separate object.
- Skeleton test: `{ role, content, ...optional }` is consistent across all
  four core sub-shapes. Provider-specific shapes (`MessageParam`,
  `ChatCompletionMessageParam`) live in the L4 sibling packages and are
  translated at the AgentAdapter boundary.
- Sentence test:
  - "AgentLoop sends Messages to AgentAdapter" — natural ✓
  - "AgentLoop sends UserMessages and SystemMessages and ToolMessages to
    AgentAdapter" — clunky and pedantic.

**Chosen name**: `Message`.

**Rejected names with reasoning**:
- `MessageParam` — leaks the Anthropic SDK's transport shape into the core
  ontology. (Vendor SDK names, like provider IDs, never enter the DOS per
  ADR-0010's port-vs-implementation separation.)
- `Conversation` — declines the offer. Per the docs Inventory's "Notable
  absences" section, the team explicitly does not introduce a
  `Conversation` aggregate. Conversations are `Message[]` carried by
  Session/AgentLoop, not a separate entity.

### Merge: `AgentLoop` ← { `AgentLoop`, `Loop`, `Agent` (orch), `IterationRunner`, `IterationCoordinator`, `IterationLifecycle`, `Run`, `AgentRun` }

**Judgment 2 application**:
- Lifecycle test:
  - `IterationRunner` / `IterationCoordinator` / `IterationLifecycle`: NO
    independent lifecycle — they are sub-modules of one `AgentLoop.run()`
    call. Per Judgment 2, "two views of one thing" — these views exist for
    code modularity, not ontology distinction. Demoted to "infrastructure
    inside AgentLoop".
  - `Agent` (in orchestration): SLIGHTLY different lifecycle — has
    registration/de-registration in addition to AgentLoop's run states.
    Strict reading: two objects. Pragmatic reading for this v0.1.0: the
    extra state is small (an AgentLoop "playing the role" of Agent in an
    Orchestrator). Recorded as Open Question Q001.
  - `Run` / `AgentRun`: ephemeral; persistent record is `Trace`.
- Permission test: identical (caller controls).
- Sentence test:
  - "AgentLoop calls AgentAdapter" — natural; matches code, README, ADR-0001.
  - "Run calls AgentAdapter" — confusing; "what runs in the run?".
  - "Agent calls AgentAdapter" — ambiguous; collides with Anthropic SDK's
    `Agent` and OpenAI's `Agent` SDK types (visible in
    `examples/shims.d.ts`).
  - "Loop calls AgentAdapter" — too generic.

**Chosen name**: `AgentLoop`.

**Rejected names with reasoning**:
- `Loop` — semantically empty; "what kind of loop" is a real question and
  the answer (an LLM-driven, tool-dispatching, guardrail-protected loop) is
  the entire point.
- `Agent` — ambiguous (overloaded with the orchestration sub-domain term
  and with vendor SDK types). The harness-one community would have to
  re-train its vocabulary to disambiguate; not worth it.
- `AgentRunner` — `Runner` is verb-suffix (Pattern 2 territory); also
  collides with `IterationRunner` internal seam.
- `Run` — would conflate the ephemeral runtime invocation with the
  long-lived AgentLoop instance; the team has explicitly decided that the
  *persistent* record of a run is `Trace`, leaving `Run` no role.

### Merge: `Trace` ← { `Trace`, `Span`, `SpanEvent`, `TokenUsage`, `ModelPricing`, `CostAlert`, `RetryMetrics` }

This is the biggest aggressive merge in the convergence pass.

**Judgment 2 application**:
- `Span` is genuinely a tree-child of Trace — same lifecycle (start/end
  within Trace's window), same eviction story (Trace eviction cascades to
  Spans). Per Judgment 2, this is the "two views of one thing" case: a
  Trace IS its Span tree, and exposing them as separate top-level objects
  doubles the rules without expressivity.
- `SpanEvent` is one rung deeper.
- `TokenUsage` is recorded *per LLM call* (i.e., per Span), aggregated up
  the Trace. Pure value object.
- `ModelPricing` is a *static* per-model rate card associated with Trace
  for cost aggregation. Reference value object.
- `CostAlert` is a derived event when cumulative cost crosses a threshold.
  Lives as data on/around the Trace.
- `RetryMetrics` is aggregated from `adapter_retry` SpanEvents — pure
  derivation.

**Chosen name**: `Trace`.

**Rejected names with reasoning**:
- `Span` (as top-level) — its lifecycle is bound to the Trace; promoting
  it would force all relationships ("Span CONTAINS SpanEvent", "Span
  HOLDS TokenUsage") that are simpler when modeled as sub-shapes of Trace.
- `Run` — ephemeral; doesn't persist.
- `Transaction` — collides with database transactions in observability
  vendor SDKs (e.g. Datadog).
- `Workflow` — ADR-0001 forbids this term in the core ontology.

### Bounded Context decisions (Judgment 4)

**Single Bounded Context for harness-one core.** Reasoning:

- All L3 subsystems share one ubiquitous language for `Message`, `Tool`,
  `Trace`, `Session`, `Guardrail`. There is no `Message` in tools/ that
  means something different than `Message` in memory/.
- The "L3 subsystems don't import each other" rule is layering, not bounded
  contexts. (The methodology doc's heuristic for splitting — different
  field sets, translation layers, different teams — fires only weakly here.)
- Sibling packages (L4: anthropic, openai, redis, ...) DO have local
  vocabularies (`MessageParam`, `ChatCompletionMessageParam`,
  `RedisClientShape`), and these are translated at the L4 boundary. The
  `AgentAdapter` interface and `MemoryStore` interface ARE the
  Anti-Corruption Layers.

**Downstream contexts** identified and listed in `bounded_contexts`:
- DevKit (`@harness-one/devkit`) — eval / evolve / drift / taste-coding.
- Application contexts (`apps/dogfood`, `apps/coding-agent`,
  `apps/research-collab`) — each has its own domain.
- Showcase context (`showcases/01..04`) — calibration projects.

**Upstream contexts**:
- LLM Provider Context (Anthropic, OpenAI, Groq, Together, ...).

This is recorded in `bounded_contexts` of the DOS and as Open Question Q004
for the DevKit relationship.

### Rules — Constitution vs. Policy (Judgment 3)

For each candidate rule found in the codebase (validation logic, invariant
tests, restrictive enums, ADR commitments, README design principles), the
constitution-vs-policy decision was:

**Kept as constitution (entered the DOS):**

- `R001` (Message has exactly one role) — without this, the message-routing
  contract collapses; downstream tool gating breaks.
- `R002` (System message trust brand on restore) — ADR-0007. Without this,
  any persisted-then-loaded "system" message is forgeable.
- `R003` (Tool definitions are frozen) — without this, the JSON-Schema
  contract for the LLM-side tool-use protocol becomes mutable mid-run.
- `R004` (Pipelines default fail-closed) — ADR-0006. Without this, the
  security posture of the entire library inverts.
- `R005` (Tool capability allow-list, default ['readonly']) — without this,
  the typical "secure preset" is no longer secure-by-default.
- `R006` (AgentLoop.run() non-re-entrant) — without this, per-run state is
  corruptible by concurrent calls; the entire iteration coordinator design
  changes.
- `R007` (traceId unifies trace/cost/token) — ADR-0005. Without this,
  observability becomes a join-on-multiple-keys problem; vendors that
  consume traces lose the correlation guarantee.
- `R008` (≥2-impl interfaces ship conformance suite) — ADR-0008. This is
  enforced by user_workflow rather than system, but it is constitutional —
  removing it would let semantic drift land unchecked.
- `R009` (stream-size hard limits in UTF-8 bytes) — ADR-0009. Without this,
  the harness silently degrades under adversarial / pathological streams.
- `R010` (tool errors as data; stack traces stripped) — ADR-0001 / README
  design principles. Without this, internal paths/types leak to LLM
  context.
- `R011` (L3 subsystems don't import each other) — ADR-0002. Without this,
  the dependency graph is no longer a layered DAG, and "where does this
  symbol live" becomes ambiguous.
- `R012` (zero runtime deps in core) — ADR-0004. Without this, the core's
  install graph balloons and Scorecard/SBOM workflows break.
- `R013` (factory functions; class stays internal) — ADR-0003. Convention
  + lint-enforced; constitutional in the API contract.

**Excluded as policy (NOT in the DOS):**

- "Default `maxIterations = N`" — pure policy; depends on use case.
  Suggested home: `AgentLoopConfig` documentation.
- "Default `maxTotalTokens = N`" — same.
- "Default rate-limiter window = N seconds" — pure policy.
- "Default `maxCallsPerTurn = 20`, `maxCallsPerSession = 100` for tool
  registry" — policy; documented in `04-tools.md` defaults but doesn't
  enter the DOS rules.
- "Default `ttlMs = 5 minutes` for SessionManager" — policy.
- "Default `maxTraces = N`" — policy.
- "Premium tiers / free tier limits" — would be policy if such a thing
  existed; harness-one is OSS, so this is non-applicable but mentioned
  for completeness.
- "Pricing tables for specific models" (which appear in

### Relationships — verb filtering

The combined verb tables from Stages 1a + 1b contained ~1,400 distinct
verbs (mostly impl verbs from code: `get`, `set`, `update`, `validate`,
`format`, `render`, `parse`, ...). After filtering for verbs that connect
two **business** objects and read naturally as `Subject VERB Object`,
the relationships section ended up with **15 relationships**.

**Notable verb decisions**:
- `CALLS` (canonical) — kept over `INVOKES` because the docs and code
  consistently use "call" for adapter calls.
- `DISPATCHES` (canonical) — kept over `EXECUTES` because the code
  separates dispatch (who's invoked) from execution (the actual run).
- `CONSUMES` / `PRODUCES` — used for the AgentLoop ↔ Message edges to
  capture both directions cleanly.
- `EMITS` — used specifically for AgentLoop → Trace (one-shot, persistent
  record).
- `ENFORCES` (subject: AgentLoop, object: Guardrail) — kept over "RUNS"
  because Guardrails are policy-enforcement, not just procedural runs.
- `RETURNS` — used for Guardrail → GuardrailVerdict; matches sentence test.
- `CONTAINS` — used for Trace → Span.
- `AGGREGATES` — used for Trace → TokenUsage; captures the rollup nature.
- `TRANSLATES` — used for AgentAdapter ↔ Message; captures the
  Anti-Corruption Layer role.
- `HOLDS` — used for Session → Message.
- `AUTHORS` — used for AgentLoop → MemoryEntry; captures the typical
  "the loop wrote this" provenance.
- `PERSISTS_VIA` — used for MemoryEntry → MemoryStore (port relationship).
- `RETRIEVES` — used for Retriever → DocumentChunk in the RAG composition.
- `CHUNKED_INTO` — passive form for Document → DocumentChunk; matches
  ChunkingStrategy's actual function name.

**Verbs rejected**:
- `CREATE` / `UPDATE` / `DELETE` (CRUD) — subsumed by lifecycle on each
  object; not modeled as separate relationships.
- `GET` / `SET` / `LIST` — accessors, not domain verbs.
- `LOG`, `REDACT`, `EXPORT` — these involve cross-cutting ports
  (Logger / Redactor / TraceExporter) that are demoted to compositions;
  the relationship between Trace and TraceExporter is captured implicitly
  in the ObservabilityStack composition.
- `SCAN` (Guardrail) — subsumed by `ENFORCES` and `RETURNS`.
- `ROUTE` (Orchestrator) — captured inside the Orchestration composition.

### Anti-patterns observed in this codebase

Patterns from `references/anti_patterns.md` plus codebase-specific examples
that entered the DOS's `anti_patterns` section. Each entry below cites the
generic Pattern and the local variant:

- **Pattern 3 (Synonym Sprawl)** — `Message` family. Preempted from
  inception via discriminated-union types in `core/types.ts`.
- **(Codebase-specific)** — Graph/Node/Edge concepts. Codified rejection
  in ADR-0001. Recorded in `agent_guidelines.must_not` AND in
  `anti_patterns` because the temptation reappears regularly in PRs.
- **Pattern 7 (Framework-Concept-as-Object)** — sibling-L3-import
  violations. Codified in `eslint.config.js` `no-restricted-imports` per
  ADR-0002. Recorded in `anti_patterns`.
- **(Codebase-specific)** — class-first APIs. Many primitives were
  originally classes (TraceManager among them) and refactored to factories
  during early hardening. Codified in ADR-0003. Recorded in
  `anti_patterns`.
- **(Codebase-specific)** — mock-based contract testing. Solved via
  `runMemoryStoreConformance`, `runRetrieverConformance`,
  `runEmbeddingModelConformance`, `runChunkingStrategyConformance`. Codified
  in ADR-0008. Recorded.
- **(Codebase-specific)** — fail-open guardrail pipelines. The default
  fail-closed is the constitutional fix. Codified in ADR-0006. Recorded.
- **(Codebase-specific)** — stack-trace leakage in tool feedback to LLM.
  Solved by structured ToolFeedback; enforced by lint rule
  `tools/lint-error-messages/check.mjs`. Recorded.

Pattern 10's ("AI-Hallucinated Concept") test.

## Caveats for the human reviewer

Things to verify before treating `dos.yaml` as ground truth:

1. **`bounded_contexts.upstream_contexts` and `downstream_contexts` are
   based on the docs' framing.** Confirm with the team that DevKit, apps,
   and showcases really are downstream contexts in a DDD sense, vs. just
   "consumers". If the team thinks of harness-one + devkit as one product,
   the bounded-context model may want to merge them.
2. **`Session` is demoted to composition, not a core object.** Some
   reviewers will disagree — see the Pattern 9 reasoning in
   `03_convergence.md` for the case it might be promoted.
3. **`audience` field is templated.** The 4 audience entries
   (developers / adapter authors / AI agents / maintainers) are inferred
   from README + CLAUDE.md; verify against the team's actual stakeholder
   list.
4. **`success_criteria` is partly inferred.** Confirm the "30 minutes"
   target is realistic for this codebase's complexity.
5. **`open_questions` has 5 entries.** All real, but the user should add
   their own from prior team discussions if any are missing. A DOS with
   too few open questions papers over real disagreements.
6. **`anti_patterns` has 7 entries** all traced to ADRs / lint rules / code
   evidence. Older anti-patterns (pre-this-codebase mistakes the team has
   seen) won't be captured — add them manually.
7. **Bounded Context detection is conservative.** The skill kept harness-one
   core as a single context. If the team is in fact splitting orchestration
   off as a separate product (or merging it back into core after a planned
   reorganization), revisit.
8. **`Run` and `Conversation` are deliberately absent.** This is recorded in
   the docs Inventory's "Notable absences" section and reflected in the
   ontology. If the team disagrees, this is a constitutional change, not a
   minor fix.
